# Skill 创建最佳实践

## 一、命名规范

### 文件夹命名
- 使用英文小写字母
- 多个单词用连字符（-）连接
- 不要用空格、下划线、中文
- 名称要能体现skill的功能

```
好的命名：
  code-reviewer
  weekly-report
  api-tester
  react-helper

坏的命名：
  CodeReviewer      ← 不要用驼峰
  weekly_report     ← 不要用下划线
  我的技能           ← 不要用中文
  tool              ← 太笼统
  my-awesome-super-cool-skill  ← 太长
```

### 文件命名
- skill.md 固定不变（必须叫这个名字）
- 其他文件用小写+连字符
- 脚本文件要有合适的扩展名（.sh, .py, .js）

## 二、skill.md 编写准则

### 2.1 触发条件：宁缺毋滥

**原则**：宁可漏掉一些触发情况，也不要误触发

```markdown
好的触发条件（精确）：
- 用户说"帮我写周报"
- 用户说"生成本周的工作总结"
- 用户明确提到要生成周报文档

坏的触发条件（太宽泛）：
- 用户提到"工作"      ← 太宽泛，很多场景都提到工作
- 用户需要帮助        ← 等于没说
- 用户写东西          ← 几乎所有任务都涉及写东西
```

### 2.2 工作流程：每步可执行

**原则**：每一步都要具体到AI能直接执行，不能有模糊描述

```markdown
好的步骤（可执行）：
步骤1：读取用户指定的代码文件
步骤2：检查是否存在未使用的变量，列出所有找到的情况
步骤3：检查函数长度是否超过50行，列出所有超长函数
步骤4：按严重程度排序，输出检查报告

坏的步骤（模糊）：
步骤1：理解代码         ← "理解"太模糊，怎么理解？
步骤2：分析问题         ← "分析"太模糊，分析什么？
步骤3：给出建议         ← "建议"太模糊，什么格式？
```

### 2.3 输出格式：有模板

**原则**：给AI一个明确的输出模板，而不是让它自由发挥

```markdown
好的输出格式定义：
  输出格式：
  ## 检查报告
  ### 严重问题
  - [文件名:行号] 问题描述
  ### 建议
  - [文件名:行号] 建议描述
  ### 总结
  共发现 X 个问题，Y 个建议。

坏的输出格式定义：
  输出一个报告    ← 什么样的报告？格式呢？
```

### 2.4 错误处理：考虑边界情况

**原则**：预想可能出错的场景，提前给出处理方式

```markdown
应该考虑的错误场景：
- 用户没有提供必要的输入 → 主动询问
- 文件不存在 → 提示用户检查路径
- 权限不够 → 说明需要什么权限
- 执行超时 → 建议缩小范围
- 格式不对 → 给出正确格式示例
```

## 三、知识文件编写准则

### 3.1 保持精简
- AI的上下文窗口是有限的
- 每个知识文件不要超过必要的长度
- 只放最核心的内容

### 3.2 结构化组织
```markdown
好的知识文件结构：
# 主题
## 子主题1
要点1
要点2
## 子主题2
要点1
要点2

坏的知识文件结构：
一大段不分段的文字......
```

### 3.3 按需加载
- 不要把所有知识都塞进 skill.md
- 拆分成多个文件，在 skill.md 中说明什么时候加载哪个
- 这样AI只在需要时才读取，节省上下文空间

## 四、工具脚本编写准则

### 4.1 安全第一
```bash
# 好的实践
set -euo pipefail          # 遇错即停，未定义变量报错
readonly WORK_DIR="./tmp"  # 限制工作目录

# 坏的实践
rm -rf /                    # 绝对不要
rm -rf $UNSET_VAR/*        # 变量未定义时会删根目录
```

### 4.2 要有参数验证
```bash
# 好的实践
if [ $# -lt 1 ]; then
    echo "用法: $0 <参数1>"
    exit 1
fi

# 坏的实践
# 不做任何检查直接用 $1
```

### 4.3 要有执行权限
```bash
chmod +x tools/your-script.sh
```

## 五、反面案例集

### 反面案例1：万能skill
```markdown
# 不要这样做
能力说明：
- 能写代码
- 能做设计
- 能写文档
- 能做测试
- 能部署
- 能做任何事
```
问题：什么都能做 = 什么都做不好。拆分成多个专注的skill。

### 反面案例2：没有触发条件
```markdown
# 不要这样做
触发条件：
- 随时可用
```
问题：AI不知道什么时候该用，可能在不该用的时候用了。

### 反面案例3：流程太粗糙
```markdown
# 不要这样做
工作流程：
1. 开始
2. 做事
3. 结束
```
问题：等于没说。AI无法按照这个流程执行任何事。

### 反面案例4：硬编码路径
```markdown
# 不要这样做
步骤1：读取 /Users/zhangsan/Desktop/my-project/src/index.js
```
问题：换个人就用不了了。使用相对路径或让用户指定路径。

## 六、质量标准速查

一个好的 Skill 应该是：

| 标准 | 说明 |
|------|------|
| 单一职责 | 只做一件事 |
| 触发精准 | 该来的时候来，不该来的时候不来 |
| 步骤清晰 | 每一步都是可执行的具体动作 |
| 格式明确 | 输出有固定模板 |
| 容错健壮 | 有错误处理方案 |
| 文档完整 | 别人拿到就能用 |
| 可维护 | 容易修改和升级 |
